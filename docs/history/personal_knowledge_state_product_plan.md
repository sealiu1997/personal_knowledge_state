# 个人知识状态库：产品规划文档

## 1. 背景：问题不是“知识库不好用”，而是“知识状态无法稳定运行”

过去几年，AI 个人知识库大多沿着同一条路线演进：把资料放进 Obsidian、Notion、RAG、Wiki 或 Agent 记忆系统里，再让模型检索、摘要、回答问题。

这些方案有用，但问题也很明显：资料越来越多，系统却不一定越来越懂用户；模型可以读到文档，却不一定知道哪些判断已经过时、哪些结论有证据、哪些内容只是临时想法，哪些约束不能被覆盖。

真正的问题不是“资料有没有被存下来”，而是：

> 一条判断凭什么能成为长期知识？  
> 谁有权把它写入系统？  
> 它什么时候失效？  
> 它能不能被追溯、修正和审计？  
> Agent 能不能基于它行动？

从认识论角度看，个人知识库不能只是资料仓库。它必须回答“什么算知道”“凭什么知道”“谁来负责事实”“哪些知识可以影响行动”。

因此，本项目不把目标定义为“再做一个更聪明的笔记软件”，而是定义为：

> 构建一个本地优先、人机共用、项目中心的个人知识状态库，让用户和各种 Agent 能围绕同一份长期知识状态协作。

---

## 2. 核心判断

### 2.1 知识库的核心不是存储，而是状态

文档、链接、笔记、收藏夹、项目文件夹都只是知识材料。真正有价值的是这些材料沉淀出的长期状态：当前目标、关键判断、设计原则、踩坑经验、决策记录、约束边界和可复用上下文。

一个可用的个人知识系统，应该能让用户在不同项目之间快速恢复状态，也能让 Agent 快速理解当前项目的目标、背景和边界。

### 2.2 Wiki、RAG、思维导图都只是投影

Wiki 适合阅读，RAG 适合检索，思维导图适合梳理结构，MCP/Skill 适合 Agent 接入。但它们都不应该成为底层事实源。

底层应该维护的是更稳定的状态对象：项目、资料、主张、证据、任务合约、权限、快照和审计记录。

### 2.3 Agent 可以提议，但不能直接负责事实

模型擅长发现关系、总结资料、提出候选判断。但模型不应该直接把自己的总结写成长期事实，更不能基于漂移后的理解修改核心架构或长期目标。

长期知识的写入应该走候选区、审核、确认和版本记录。

### 2.4 权限和约束不能只靠 Prompt 或 Markdown

Prompt 和 Markdown 可以解释规则，但不能执行规则。

真正可靠的系统必须有硬的软件架构：Agent 不直接接触长期状态，所有写入和变更都通过内核、任务合约、权限策略、工作区、补丁、审核和审计流程完成。

### 2.5 模板的目标是减负，不是增加仪式

系统应该帮助用户更轻松地开启一个项目，而不是要求用户先学习一套复杂知识管理方法。

模板必须精简、必要、可选配。不同类型项目可以有不同模板，但所有模板都应该服务于同一个目标：降低启动成本，快速生成可用的项目状态和 Agent 上下文。

---

## 3. 产品定位

本产品是一个本地优先的个人知识状态库，面向个人长期项目、研究、写作、软件开发、游戏开发和 Agent 协作场景。

它不是：

- 另一个 Notion；
- 另一个 Obsidian；
- 另一个 RAG 数据库；
- 另一个全自动 AI 记忆系统；
- 另一个复杂项目管理工具。

它是：

> 一个围绕项目组织长期知识状态，并通过受控接口提供给人类和 Agent 共同使用的知识底座。

用户可以用它来管理：

- 项目目标；
- 项目资料；
- 过程文档；
- 技术选型；
- 设计原则；
- 个人判断；
- 踩坑记录；
- 写作素材；
- 重要链接；
- Agent 上下文；
- 项目变更历史。

---

## 4. 产品形态

产品的核心形态不是一个单一 App，而是一个本地知识内核加多种接口。

底层是本地文件和数据库；中间是知识状态内核；上层可以接 CLI、Web UI、MCP、RAG、Wiki、Skill、Agent 插件或编辑器扩展。

### 4.1 人类视图

人类主要面对的是文件夹、Markdown、项目主页、状态页、资料页和审核队列。

用户应该能用普通编辑器、Obsidian、VS Code 或未来的 Web UI 直接查看和编辑大多数内容。

### 4.2 Agent 视图

Agent 不应该直接面对一整个混乱 vault，而应该通过受控接口读取：

- 当前项目上下文；
- 任务合约；
- 已接受判断；
- 关键资料；
- 当前限制；
- 可执行范围；
- 候选写入入口。

### 4.3 系统视图

系统内部维护的是：

- Project Capsule；
- Template；
- Source；
- Claim；
- Evidence；
- Task Contract；
- Policy；
- Workspace；
- Patch；
- Candidate；
- Snapshot；
- Audit Log；
- Context Pack。

## 5. 核心架构

### 5.1 Project Capsule：项目胶囊

Project Capsule 是系统的基本组织单元。

每个项目胶囊代表一个可长期维护的项目状态。它可以是一个软件项目、游戏项目、文章项目、视频项目、研究项目，也可以是一个长期学习主题。

最小结构包括：

- `project.yaml`：机器可读的项目元信息；
- `README.md`：项目是什么、为什么做、交付物是什么；
- `state.md`：当前阶段、当前目标、下一步；
- `guardrails.md`：长期目标、非目标、禁止事项、需要确认的事项；
- `context.md`：给 Agent 使用的项目上下文包。

其他模块都应选配，例如：

- `sources.md`；
- `claims.md`；
- `decisions.md`；
- `lessons.md`；
- `architecture.md`；
- `outline.md`；
- `style.md`；
- `facts.md`；
- `assets.md`；
- `tasks.md`。

项目胶囊的目标是让项目状态可读、可迁移、可投影、可被 Agent 使用。

---

### 5.2 Template Engine：模板引擎

模板引擎负责帮助用户低成本开启新项目。

它不追求完整，而追求必要。创建项目时，系统只应询问少量关键问题：

1. 这个项目是什么？
2. 最终交付物是什么？
3. 当前阶段是什么？
4. 是否已有资料或仓库？
5. Agent 默认权限是什么？

不同项目类型使用不同模板：

- 文章项目：目标、读者、核心观点、资料、风格、大纲；
- 视频项目：主题、脚本、素材、风格、事实来源；
- 软件项目：仓库、架构、环境、任务、技术决策；
- 游戏项目：玩法、版本目标、资源规范、技术架构、世界观；
- 研究项目：研究问题、资料、术语、当前判断、待验证问题。

模板系统必须遵守一个原则：

> 允许粗糙启动，逐步结构化。

用户不应该为了开启一个项目而填写大量字段。系统应该先生成最小可用结构，再在项目推进过程中沉淀更多模块。

---

### 5.3 Context Pack Generator：上下文包生成器

Context Pack 是 Agent 使用项目状态的主要入口。

它把项目胶囊中的信息压缩为适合当前 Agent 和任务的上下文包，包括：

- 项目简介；
- 当前目标；
- 当前阶段；
- 关键资料；
- 已接受判断；
- 重要决策；
- 踩坑经验；
- 当前任务；
- 非目标；
- 权限边界；
- 需要人类确认的事项。

Context Pack 可以输出为：

- Markdown；
- JSON；
- MCP Resource；
- Agent-specific prompt；
- 可复制文本。

它的作用不是让 Agent “知道一切”，而是给 Agent 一份足够小、足够明确、足够受控的项目脑。

---

### 5.4 PKS Kernel：知识状态内核

PKS Kernel 是系统的硬内核。

它负责管理项目状态、权限、任务、写入、候选、补丁、快照和审计。Agent 不应该直接修改长期知识库，而应该通过 Kernel 提交操作请求。

Kernel 的主要组件包括：

- Project Registry：项目注册表；
- Task Contract Engine：任务合约引擎；
- Policy Engine：权限策略引擎；
- Workspace Manager：任务工作区管理；
- Patch Manager：补丁生成与合并；
- Candidate Queue：候选写入队列；
- Snapshot Manager：快照管理；
- Audit Log：审计日志；
- Context Engine：上下文包生成；
- MCP / CLI / API Adapter：外部接口。

Kernel 的核心原则：

> Agent 不直接接触长期状态。  
> 所有操作必须通过受控 API。  
> 所有写入必须绑定任务、权限和审计记录。

---

### 5.5 Task Contract：任务合约

Task Contract 用于防止目标漂移。

每次 Agent 执行一个任务前，都应该绑定一个明确的任务合约。任务合约说明：

- 当前任务是什么；
- 为什么做；
- 允许修改什么；
- 禁止修改什么；
- 哪些操作需要确认；
- 验收标准是什么；
- 触发停止的条件是什么。

例如：

- 允许修改 `templates/article/**`；
- 禁止修改 `PKS_MANIFEST.md`；
- 修改权限策略必须请求确认；
- 新模板初始化问题不能超过 5 个；
- 如果需要改变核心架构，停止并提交架构变更提案。

Task Contract 不是 Prompt 装饰，而是 Kernel 执行权限判断的依据。

---

### 5.6 Policy Engine：权限策略引擎

权限系统是防止 Agent 漂移和误操作的关键。

权限不应只按角色划分，而应同时考虑：

- 谁在操作；
- 操作什么对象；
- 执行什么动作；
- 当前任务合约是否允许；
- 目标文件是否受保护；
- 当前项目状态是否允许变更。

典型权限层级：

- 只读模式：只能搜索、读取、总结；
- 草稿模式：可以写入 `drafts/`、`candidate/`、`inbox/`；
- 项目工作模式：可以在允许范围内修改非核心文件；
- 补丁模式：可以提交对核心文件的 patch，但不能直接合并；
- 全权模式：MVP 阶段不支持。

受保护对象包括：

- `PKS_MANIFEST.md`；
- `project.yaml`；
- `guardrails.md`；
- 核心模板；
- 权限策略；
- accepted claims；
- snapshots；
- audit log。

Agent 对这些对象默认只能读取或提交候选修改。

---

### 5.7 Workspace / Patch / Candidate：受控写入链路

Agent 的写入不应直接进入主知识库。

标准流程是：

1. Agent 读取 Context Pack；
2. Kernel 创建任务工作区；
3. Agent 在工作区中生成草稿、候选主张或补丁；
4. Policy Engine 检查是否越权；
5. Patch Manager 生成变更集；
6. Human Review 或规则审核；
7. 通过后合并到主知识库；
8. 生成 Snapshot 和 Audit Log。

这样即使 Agent 在长上下文中发生目标漂移，也只能污染工作区，不能直接污染长期状态。

---

### 5.8 Snapshot / Audit：快照与审计

长期项目必须能回答：

> 当时为什么这么做？  
> 当时基于哪些判断？  
> 谁改了什么？  
> 哪次变更改变了项目方向？

因此，系统需要在关键节点生成快照：

- 项目创建；
- 阶段切换；
- 核心目标变化；
- 重要决策确认；
- 核心模板变化；
- 权限策略变化；
- 大型 Agent 任务合并。

审计日志记录每次关键操作，包括 actor、task、operation、target、decision、reason 和 timestamp。

## 6. 技术栈建议

MVP 采用本地优先、轻量、可迁移的技术栈。

建议组合：

- 文件资产：Markdown + YAML + Git；
- 状态数据库：SQLite；
- 全文搜索：SQLite FTS；
- 向量检索：LanceDB 或 Qdrant，本地优先；
- 后端服务：Python + FastAPI；
- CLI：Typer；
- Schema：Pydantic；
- Agent 接入：MCP Server；
- 人类编辑：VS Code / Obsidian / 文件系统；
- 未来 UI：轻量 Web UI。

技术选型原则：

> 文件是长期资产，数据库是索引和运行时。  
> 接口可以替换，核心状态不能被平台锁死。

## 7. MVP 范围

MVP 不追求完整知识图谱，也不追求全自动整理。MVP 的目标是验证一个最小闭环：

> 用户能快速创建项目，生成项目上下文，让 Agent 在受控边界内协作，并把有价值的结果沉淀回长期状态。

### 7.1 MVP 必做

- `pks new`：创建项目胶囊；
- `pks context`：生成 Agent 上下文包；
- `pks task`：创建任务合约；
- `pks candidate`：提交候选写入；
- `pks snapshot`：生成项目快照；
- `pks search`：搜索项目资料；
- MCP Server：提供只读和候选写入接口；
- Policy Engine：保护核心文件和长期状态；
- Workspace：Agent 写入沙盒；
- Patch Review：核心变更以补丁形式提交。

### 7.2 MVP 不做

- 完整 Web UI；
- 移动端；
- 多用户协作；
- 全自动知识整理；
- Agent 全权写入；
- 复杂知识图谱；
- 自动接受长期主张；
- 复杂工作流编排平台。

## 8. 典型流程

### 8.1 创建一个文章项目

用户输入：

```bash
pks new article "加密技术科普文章"
```

系统只询问少量问题：

- 文章想回答什么问题？
- 目标读者是谁？
- 当前核心观点是什么？
- 已有资料在哪里？
- 是否有风格要求？

系统生成：

- README；
- state；
- outline；
- sources；
- claims；
- style；
- guardrails；
- context。

随后用户可以把 `context.md` 交给任意 Agent，让它在明确边界内协助写作。

### 8.2 创建一个软件项目

用户输入：

```bash
pks new software "个人知识状态库"
```

系统生成：

- 项目目标；
- 当前阶段；
- Git 仓库入口；
- 架构文档；
- 任务列表；
- Agent 修改边界；
- 初始上下文包。

Agent 如果要修改核心架构，不能直接改文件，只能提交架构变更提案或 patch。

### 8.3 Agent 协作流程

1. 用户创建 Task Contract；
2. Agent 读取项目 Context Pack；
3. Agent 在 Workspace 中工作；
4. Agent 提交候选笔记、候选主张或 patch；
5. Kernel 检查权限和范围；
6. 人类审核；
7. 通过后合并；
8. 系统生成快照和审计记录。

## 9. 成功标准

MVP 成功不以“功能多”为标准，而以是否真的降低心智负担为标准。

关键指标：

1. 新项目能否在 5 分钟内启动；
2. 用户切换项目时能否快速恢复状态；
3. Agent 是否能通过 Context Pack 快速进入项目；
4. Agent 是否减少了重复询问背景信息；
5. Agent 是否无法直接越权修改核心状态；
6. 重要判断是否能进入候选区，而不是污染长期知识；
7. 用户是否愿意在真实项目中持续使用；
8. 系统是否比普通文件夹、收藏夹和笔记更省心。

---

## 10. 设计原则总结

1. 项目优先，而不是资料优先。
2. 状态优先，而不是文档优先。
3. 模板减负，而不是增加仪式。
4. 文件是长期资产，数据库是运行时。
5. Context Pack 是 Agent 的主要入口。
6. Guardrails 和 Task Contract 是防漂移核心。
7. Agent 默认不能直接修改长期状态。
8. 所有写入必须经过 Kernel、Policy、Workspace 和 Review。
9. Wiki、RAG、MCP、Skill 都是接口和投影，不是事实源。
10. 系统应允许粗糙启动，逐步结构化。

---

## 11. 最终产品主张

个人知识库真正的问题，不是资料不够多，也不是检索不够强，而是长期知识状态无法稳定运行。

本产品试图解决的不是“如何让 AI 记住更多”，而是：

> 如何让人类和 Agent 围绕同一份可追溯、可审计、可投影、可授权的项目知识状态协作。

它的最小原型不是一个全能知识管理系统，而是一个本地知识状态内核：

- 用项目胶囊组织知识；
- 用模板降低启动成本；
- 用上下文包帮助 Agent 接入；
- 用任务合约防止目标漂移；
- 用权限内核约束写入；
- 用候选区和补丁机制沉淀长期知识；
- 用快照和审计保留历史。

如果这个闭环跑通，系统就不只是“更聪明的资料夹”，而会成为用户在长期项目中反复使用的个人知识底座。
