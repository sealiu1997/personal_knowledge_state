# 从个人知识库到职业 Agent：长期知识状态、职业模型与工程边界研究报告

## 摘要

过去几年，“AI 个人知识库”反复以不同形态出现：Notion 加 ChatGPT、RAG 加 Obsidian、Obsidian 加 Claude Code、带长期记忆的 agent，以及 Karpathy 提出的 LLM Wiki。每一轮方案都会让人短暂觉得问题终于被解决了，但用到一定规模后，常见故障又会重新出现：旧文档被当成新事实，模型总结覆盖了曾经确认过的判断，长期偏好被误写，安全约束在上下文压缩中消失，系统说着说着发生行为漂移。

这些问题表面上属于工程问题，深处却是认识论问题：一条判断凭什么被系统当作知识？谁有权把模型总结升级成长期事实？当新证据和旧判断冲突时，系统如何处理？当知识会影响工具行动时，事实判断和行动授权应该如何分开？

李自然的原文把“AI 个人知识库”的问题从存储和检索推进到“长期知识状态”的维护。随后的问答进一步把这个问题扩展到“职业 agent”：如果 AI 要替代某个职业的一部分，仅靠通用模型、RAG 和 workflow 不够；职业直觉、规范意识和风险判断必须部分内化到专有模型中，但动态事实、长期状态、权限边界和审计责任仍然必须留在外部系统。编程 agent 是这个方向目前最成熟的样本，也最清楚地暴露出这套架构的瓶颈。

本文的核心结论是：未来可用的职业 agent 既不是纯模型路线，也不是纯工程路线，而是“厚模型 + 动态知识系统 + 长期状态运行时 + 硬权限边界 + 审计机制”的复合系统。模型负责提出判断和执行专业推理，系统负责事实权威、状态稳定、风险分级和行动约束。

## 一、问题的真正起点：知识库不是知识状态

很多关于 AI 个人知识库的讨论之所以混乱，是因为“记忆”一词承载了太多不同层次的东西。李自然原文首先做了一个重要区分：文档知识库、会话上下文管理、智能体记忆运行时、个人知识状态系统，这四者都和“记住”有关，但解决的问题并不相同。

文档知识库解决的是资料放在哪里、怎么搜出来。会话上下文管理解决的是一场对话太长以后，怎样摘要、压缩和裁剪。智能体记忆运行时解决的是一个长期 agent 什么时候读记忆、什么时候写记忆、什么时候审核和冻结状态。个人知识状态系统则更进一步，它关心一个人、一个项目或一个领域在多年尺度上如何保持知识的准确性、可追溯性和可更新性。

当前许多“AI 个人知识库”的失败，正是因为试图用第一层工具解决第四层问题。RAG 和 Obsidian 可以帮助模型查资料，但它们本身并不维护知识。它们不知道哪份文档过期，不知道两份材料是否冲突，不知道一个判断是否被后来的证据推翻，也不知道某个结论具体来自哪段原文。

因此，真正的问题不是“如何让 AI 搜到更多资料”，而是“如何让一份长期知识状态在 AI 系统中稳定运行”。这意味着系统不只要保存文本，还要保存判断的来源、证据、时间、状态、风险等级和更新关系。

## 二、认识论视角：AI 系统怎样才算“知道”

问答一开始从哲学认识论切入。认识论关心的不是某个结论本身，而是人凭什么说自己知道这件事。一个判断要成为知识，通常至少涉及信念、真实性和正当理由。把这个框架放到 AI 系统中，就会得到一组更工程化的问题：

一条判断是从哪里来的？它是原始材料直接声称的，还是模型推断出来的？证据是强支持、弱支持、间接相关，还是只说明某人在某时说过这句话？这条判断是否有反证？它在什么时间范围内有效？它能不能覆盖旧判断？它能不能影响工具动作？

这正是李自然原文中“模型可以提议，但谁来负责事实”的核心。模型可以读材料、发现关系、提出候选主张，但它不能同时扮演提议者、裁判者和行动授权者。如果模型自己总结、自己写入长期记忆、自己批准自己的结论，那么系统就把认识权、记忆权和行动权混在了一起。

从认识论角度看，长期知识系统至少要区分五层东西：原始记录、观察事实、内容主张、系统判断和长期知识。原始记录只能说明某段资料存在；观察事实只能说明某人在某时说过或写过某句话；内容主张是这句话声称的事情；系统判断是系统认为这个主张有多可信；长期知识则是经过审核后被接受为当前有效状态的判断。

例如，用户说“我以后都不喝咖啡了”，系统不能直接记成“用户不喝咖啡”。更严谨的记录应该是：用户在某个时间表达过不再喝咖啡的意向，这可能代表新的偏好，但需要后续行为确认。这个例子说明，追到原文只是最低要求，它只能证明判断不是凭空生成的，并不能证明判断本身是真的。

## 三、李自然原文的核心模型：模型提议，系统负责权威

李自然原文对几类主流方案做了分层评价。

RAG 和 Obsidian 的问题是只能查，不能积累。它们每次回答都重新临时综合，今天得到的关系不会自然成为明天的知识状态。

让模型自己沉淀记忆的方案更进一步，但风险也更大。模型既负责发现候选记忆，又负责决定什么可以成为长期事实，这会导致关键约束被摘要丢失、偏好被误写、行动权限被误解。文中提到的邮件事故说明，安全约束、审批合约、长期偏好、临时任务和会话摘要不应该混在同一条可压缩上下文链里。

Hermes 这类记忆运行时比普通记忆功能成熟得多。它通过会话开始时冻结记忆快照、异步处理记忆读写、把 review 从主回答链路中拆出去、抽象不同 memory provider 等方式，解决了长期 agent 的工作记忆稳定性问题。但它仍主要是 agent 级工作记忆，而不是多年尺度的个人知识状态系统。它解决了“记忆什么时候进入模型”，但没有充分解决“什么东西有资格成为事实”。

Karpathy 的 LLM Wiki 则把问题从“使用时临时综合”推进到“资料进入时提前综合”。这比 RAG 更像知识积累，但它的危险在于：如果后续查询主要读 Wiki，而不是强制回到原始材料和证据链，Wiki 就可能把一次有损综合固化成长期真相。维基越完整，就越像答案；越像答案，使用者就越不愿回到原文。于是派生内容反过来冒充事实源。

李自然给出的方向是：原始资料进入系统后，先变成可追溯的观察；模型基于观察提出候选主张；主张绑定证据、时间和风险等级；低风险内容可以自动接受，中高风险内容进入审核；通过后的主张进入长期状态；维基、报告、上下文包和知识图谱都只是从同一份状态投影出来；危险动作则不走记忆链路，而走独立权限审批。

在运行时层面，这要求系统至少具备四种能力：统一的记忆边界、冻结快照、候选门和版本可观测性。统一边界意味着所有记忆读写、prompt 注入、后台任务和清理动作都由专门层负责，而不是散落在不同工具和提示词中。冻结快照保证同一场对话看到的是同一版状态。候选门防止模型把一次性上下文、弱证据推断或高风险用户画像直接写成长期事实。版本与可观测性则让用户能够追问：当前对话用了哪份快照，哪些主张被召回，哪些内容还是候选，哪些证据支持它，哪些后台任务正在运行。

这个模型可以概括为两句话：模型负责提议，系统负责权威。模型可以说“我认为资料说明了 X”，系统则必须追问“证据是哪一段、推理关系是什么、有没有反证、什么时候有效、能否进入长期状态、能否影响工具动作”。

## 四、这套模型仍需补强的地方

问答对李自然原文的评价是：方向正确、基本自洽，但仍有几处未完成。

第一，原始资料不等于真相。邮件、会议纪要、用户笔记、网页资料、草案文档都可能包含误解、过时信息、反讽、假设或临时情绪。因此系统不能只记录“证据在哪里”，还要记录证据的类型、来源角色、可信度、制度地位和可能的利益冲突。这涉及社会认识论：同一句话来自医生、用户猜测、营销材料、临床指南或模型总结，其认识论地位完全不同。

第二，“主张”是合理的底层单位，但粒度很难控制。主张太粗就不可验证，主张太细又会让系统爆炸。因此需要主张类型规范，例如事实型主张、来源声称型主张、系统推断型主张、用户偏好型主张、规则型主张、权限型主张和预测型主张。不同类型应采用不同证据标准、审核策略和过期机制。

第三，证据不应该只是 citation，而应该成为论证关系。证据可能直接证明、间接支持、只在某条件下成立、只支持主张的一部分，也可能构成反证或后来推翻旧判断。更理想的结构不是“Claim A supported_by Evidence B”，而是“证据在某条件下，通过某种推理方式，对主张的某个部分提供某种强度的支持，同时是否存在反证和失效条件”。

第四，长期知识不应该只有 true/false。更合理的是维护认识状态，例如 accepted、tentative、contested、superseded、expired、rejected、unknown 或 policy-bound。这样系统既能表达“当前接受”，也能表达“暂时接受”“有争议”“已被替代”“过期失效”“证据不足”以及“这不是事实而是规则/约束”。

第五，冻结快照是好设计，但不能绝对化。冻结快照能保证同一场对话中的认知稳定性，避免模型说着说着换了一套记忆。但如果当前快照包含错误，而后台已经发现高优先级修正，系统就需要“快照隔离 + 显式升级”机制：默认不改变当前会话状态，但遇到安全约束、权限变化或关键事实更新时，提示用户是否切换到新状态。

第六，不是所有知识都适合 claim 化。事实、规则和权限可以结构化为主张，但写作风格、审美偏好、工作习惯、项目手感和用户思维方式等软模式，很难完全变成明确命题。这类知识应默认弱化、可见、可编辑、可撤销，不能像事实一样强写入，否则系统会形成隐蔽且难以察觉的刻板用户画像。

第七，审核负担可能成为产品失败点。如果每一百条候选主张都需要用户人工审核九十条，用户很快会放弃。长期知识系统必须同时解决可靠性和可用性：既不能让模型乱写，也不能把人淹没在审核工作里。

第八，用户需要知识审计界面。用户应该能看到系统认为自己有哪些长期偏好，哪些来自明确表达，哪些来自模型推断，哪些证据支持它，哪些可以撤销，哪些会影响 agent 行动。否则长期记忆会变成黑箱，用户只能看到结论，却看不到结论如何形成。

## 五、从个人知识库到职业 Agent：问题开始转向模型能力

问答后半段的重要转折是：如果所有职业规范、操作技巧、判断标准和风险边界都外包给工程规则，系统会变得非常复杂、脆弱且难以维护。此时问题不再只是工程问题，而变成模型能力问题。

一个真正能替代部分职业劳动的系统，不应该每次都靠查手册工作。资深律师、医生、投研分析师、产品经理或软件架构师之所以有职业能力，不只是因为他们掌握了大量资料，而是因为他们内化了判断框架：知道什么问题重要，什么证据可靠，什么说法有坑，哪些风险不能碰，哪些细节值得追问，哪些方案看起来能跑但长期会出事。

这些能力很难完全靠 RAG 检索出来。RAG 适合处理动态事实，例如最新法规、新闻、财报、论文、公司文档、项目状态、客户资料、会议纪要和版本信息。但 RAG 不适合承担职业判断，例如如何识别异常模式、如何权衡风险、如何判断证据强度、如何在不完整信息下做专业决策。

因此，更合理的分工是：RAG 管“现在有什么事实”，专有职业模型管“这些事实意味着什么”。但专有模型也不能取代工程层。模型权重中的知识不可审计，更新成本高，旧知识撤销困难，也不能承担权限和责任。因此，职业 agent 的合理架构不是“模型 vs 工程”的二选一，而是分层协作。

这套分层包括：基础模型、专有职业模型、外部知识系统、权限执行系统和审计机制。基础模型提供语言、推理和常识迁移能力；专有职业模型内化行业判断、规范意识、常见模式和风险感；外部知识系统维护实时事实、组织状态和历史材料；权限系统决定哪些动作能做、谁批准、如何回滚；审计系统负责解释、追踪和追责。

这里还需要强调一点：职业直觉不能只靠行业语料继续训练。只用专业文档训练出来的模型，可能更会说行业黑话，但未必更会工作。真正的职业能力来自反馈结构：专家决策轨迹、人类修改记录、成功和失败案例、复盘材料、反事实样本、长周期结果数据、高质量偏好数据和对抗性评测。职业模型要学的不是“这个行业怎么表达”，而是“这个行业如何判断”。

## 六、编程 Agent：职业 AI 的最成熟样本

编程 agent 是目前最接近职业 AI 的形态。它天然具备上述几层结构：强基础模型对应 GPT、Claude 等基模；专业训练和对齐对应代码生成、调试、补丁、终端操作和代码审查能力；仓库、README、设计文档、测试、issue 和 commit history 构成外部知识材料；沙盒、确认执行、自动模式、PR review 和 CI 构成相对成熟的权限与验证环境。

但正因为编程 agent 已经相对成熟，它暴露的问题也最有研究价值。

首先，代码仓库还不是知识系统。现在很多 agent 的知识系统只是“让模型读文件”。但复杂代码库不是文件集合，而是运行中的知识结构。成熟的 repo knowledge system 应该知道模块依赖、核心路径、历史包袱、接口边界、测试覆盖、过期文档、设计决策来源、性能敏感区、安全敏感区和下游影响。仓库只是知识材料，不是知识状态。

其次，上下文不够并不只是 token 问题，而是知识投影问题。即使上下文窗口足够大，把整个仓库塞给模型也不是正确方式。不同任务需要不同的工作包：bugfix 需要问题描述、调用链、可疑模块、相关测试和最近变更；重构需要模块边界、外部调用者、兼容性和迁移步骤；新功能需要产品意图、现有架构、相似实现、数据模型、API 约定和测试计划。知识系统的价值不是把所有东西给模型，而是把仓库投影成与当前任务相关、结构化、可验证的上下文包。

再次，review 正成为编程 agent 的瓶颈。AI 提高了代码生产速度，但没有同步提高验证速度。代码产量增加后，diff 变大，reviewer 更累，隐性 bug 和技术债可能更快积累。代码 review 不只是看能否编译和测试通过，还要判断是否符合需求、是否符合架构、是否可维护、是否存在安全、性能和边界条件问题。未来的编程 agent 不只是写代码 agent，还需要内化更多 review 能力，同时由工程层提供硬验证和风险分级，让人类重点审核高风险、高不确定性、高影响面的修改。

第四，目的漂移是编程 agent 的核心问题。模型可能发生需求漂移、实现漂移、抽象漂移、局部最优漂移和验证漂移。用户原本只要求增加一个小功能，模型可能顺手改数据结构、重构模块或引入新抽象；模型在终端循环中也可能从解决原问题变成追逐一个又一个报错；更危险的是，模型可能从“实现正确行为”漂移成“让测试通过”，甚至弱化测试来适配代码。

解决目的漂移不能只靠 prompt，而需要计划合约。计划合约应明确目标、非目标、允许修改范围、禁止修改范围、兼容性要求、测试要求、验收标准、风险等级和需要确认的动作。每当 agent 要偏离计划时，它必须显式说明原因、影响和新计划，并请求确认。这样，意图不再只是聊天记录中的一句话，而成为可审计、可比较、可执行的状态对象。

第五，模型存在上下文甜蜜点。上下文太短，模型不了解项目；上下文刚好，模型能保持高质量；上下文太长，注意力被稀释；对话太久，会积累中间尝试、失败路径和临时假设；终端循环太长，模型可能陷入局部修补。成熟的编程 agent 需要按阶段管理认知状态：规划阶段需要需求、架构和验收标准；定位阶段需要调用链、日志和历史变更；编码阶段需要局部代码和接口定义；调试阶段需要测试结果和假设列表；review 阶段需要最终 diff、原始需求、计划合约、风险清单和验证结果。

因此，理想的编程 agent 可能不是单一 agent，而是多状态机：Planner 负责计划合约，Retriever 或 Repo Cartographer 负责仓库地图，Implementer 负责范围内编码，Tester 负责测试与失败解释，Reviewer 对照计划审查 diff，Architect 守住架构边界，Permission Layer 决定哪些动作必须人类批准。长期看，这些角色不一定都以外部 agent 形式存在，也可能由更强模型内化，但关键检查点仍应保留在外部系统中。

## 七、对其他职业 Agent 的启示

编程是 AI 最有优势的职业场景之一，因为代码可以运行，编译器会报错，测试可以自动化，git 可以回滚，diff 可以审查，沙盒可以隔离，CI 可以持续检查。即便如此，编程 agent 仍然面临仓库知识不完整、上下文不足、review 跟不上、目的漂移、架构理解不足和长任务状态波动等问题。

这对法律、医疗、投研、管理咨询、教育等职业场景有重要警示：如果在代码这种反馈相对明确、验证手段相对成熟的领域，模型仍不能只靠生成能力替代完整职业系统，那么在反馈更慢、责任更重、标准更模糊、结果更难验证的领域，职业 agent 会更依赖知识状态、权限边界和审计机制。

由此可以得到一个更一般的结论：职业 AI 的核心不是“更会生成”，而是能否持续维护意图、证据、状态、质量和责任。写代码不是软件工程的全部；同样，生成法律意见、医疗建议或投资分析也不是职业实践的全部。职业实践还包括识别约束、管理风险、维护上下文、接受反馈、更新判断和承担责任。

## 八、结论：模型负责会做事，系统负责不乱来

从李自然的 AI 个人知识库原文，到后续关于职业 agent 和编程 agent 的问答，问题主线其实是一致的：AI 系统真正难的不是存储资料，也不是生成答案，而是维护一份可验证、可更新、可审计、可授权的长期知识状态。

个人知识库的问题暴露了“事实权威”不能交给模型自我总结。职业 agent 的问题进一步说明，“职业直觉”也不能完全外包给工程脚手架。成熟系统必须把能力放在正确位置：职业判断、规范意识和风险感要尽量进入专有模型；实时事实、组织状态和历史材料要放在外部知识系统；长期结论、用户偏好和项目状态要进入可追溯的状态运行时；危险动作、审批和责任要交给独立权限系统；最终还要通过审计机制解释系统为什么这样判断、这样行动。

这不是纯工程派，也不是纯模型派。纯工程派会造出复杂脆弱、维护成本极高的系统；纯模型派会造出看起来聪明但事实不可追、权限不清、出错难查的系统。更现实的路线是厚模型配上薄而硬的工程边界：模型越来越会做事，系统始终负责不乱来。

评估这样的系统，也不能只看它回答得漂不漂亮。更重要的是看它能否追到证据，能否发现冲突，能否让过期判断失效，能否在同一场对话中保持状态稳定，能否控制审核负担，能否保证不同视图来自同一份状态源，能否在高风险动作前守住权限边界。

最终，个人知识库和职业 agent 都在逼近同一个问题：AI 不是只需要更多记忆，而是需要一种可治理的认识结构。只有当系统能清楚地区分资料、观察、主张、判断、状态和授权时，它才不只是一个更聪明的资料夹，也不只是一个更快的执行器，而开始接近一个可以长期协作的职业系统。
