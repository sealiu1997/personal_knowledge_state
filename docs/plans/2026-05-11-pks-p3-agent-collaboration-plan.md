# English Version

## Purpose

P3 enables **Agent collaboration** through a controlled MCP interface and completes the **Web UI management interface** for human review and administration.

P2 delivered automated maintenance and a minimal Web UI. P3 should:
- Let Agents query and submit knowledge through MCP (standard protocol)
- Give humans a complete management interface (batch review, Claim creation, multi-dimension browsing, config management)
- Implement minimal MCP auth (read/write token, managed by MCP module, displayed in Web UI)

## 1. P3 Scope

### Core deliverables

- **MCP Server**: expose Kernel read/write methods as MCP tools
- **MCP Auth**: token-based read/write permission (token generated/managed by MCP module, displayed in Web UI with regenerate/copy)
- **Web UI completion**: batch review, Claim creation form, multi-dimension Claim browsing, Claim editing, evidence chain visualization, config management, MCP token display
- **Kernel token permission**: token permission validation lives in Kernel (not in MCP adapter)

### Not in P3

- Task Contract Engine (PKS is not a project management tool)
- Per-agent fine-grained permissions (MVP only needs token-level read/write split)
- Multi-user collaboration
- Obsidian/Notion integration (future adapter work)
- SQLite index (add when query performance requires it)
- Scheduled maintenance daemon

## 2. Decisions

- MCP Server is a thin Kernel adapter. No business logic.
- Agent behavior is constrained by Kernel's existing mechanisms (min_support, review, conflict detection). MCP does not add a separate policy layer.
- MCP auth: read tools are open, write tools require a valid token.
- Token generation/revocation is managed by MCP module. Token permission validation is in Kernel.
- Web UI displays tokens (regenerate/copy buttons) but does not manage token logic.
- Web UI Claim creation goes through `submit_candidate` (reuses existing review flow).
- Web UI Claim editing goes through `patch_projection_claim` (semantic → candidate, non-semantic → direct).
- Claim ID is auto-generated by Kernel; Web UI does not expose claim_id input.
- No new heavy dependencies beyond MCP SDK.

## 3. MCP Server (Step 1)

### 3.1 Tools

**Read (no token required):**

| Tool | Kernel Method | Description |
|------|---------------|-------------|
| `get_project_context` | `render_context` | Return PKS.md content |
| `search_claims` | `list_claims` | Search with filters |
| `get_claim` | `load_claim` | Single Claim detail |
| `get_health` | `health_check` | Health report |
| `list_projects` | `list_capsules` | Project list |

**Write (token required):**

| Tool | Kernel Method | Description |
|------|---------------|-------------|
| `submit_candidate_claim` | `submit_candidate` | Submit candidate (Kernel validates min_support) |

### 3.2 Auth Architecture

```text
Token lifecycle:
  MCP module generates token → stores hash in ~/.pks/config.yaml
  Human copies token from Web UI → provides to Agent
  Agent sends token with MCP write request
  MCP module extracts token → Kernel validates permission
  Kernel checks token hash against config.yaml → allow/deny
```

Responsibility split:

| Concern | Owner |
|---------|-------|
| Token generation/revocation | MCP module (`pks mcp token create/revoke`) |
| Token hash storage | `~/.pks/config.yaml` (Kernel-accessible) |
| Token permission validation | Kernel |
| Token display (regenerate/copy) | Web UI (calls MCP module interface) |

### 3.3 Module Structure

```text
src/pks/mcp/
├── __init__.py
├── server.py          # MCP Server (mcp SDK)
├── tools/
│   ├── __init__.py
│   ├── read.py        # Read-only tools
│   └── write.py       # Write tools (token-gated)
├── auth.py            # Token generation, revocation, hash management
└── config.py          # MCP config
```

### 3.4 Kernel Addition

New Kernel method for token permission validation:
- `validate_token(token_hash) -> TokenPermission` (read/write/invalid)

Token permission data stored in `~/.pks/config.yaml`:
```yaml
mcp_tokens:
  - token_id: "tok_001"
    token_hash: "sha256:..."
    created_at: "2026-05-11T10:00:00+08:00"
    label: "Claude Code workspace"
    permissions: [read, write]
```

### 3.5 CLI

```bash
pks mcp start                              # Start (default stdio)
pks mcp start --transport stdio            # For Claude Code, Codex
pks mcp start --transport sse              # For web Agents
pks mcp token create --label "Claude Code" # Generate token (prints once)
pks mcp token list                         # List tokens (label + permissions, no plaintext)
pks mcp token revoke <token_id>            # Revoke token
```

## 6. Acceptance Criteria

### MCP Server
- `pks mcp start` launches MCP server with stdio transport.
- Read tools work without token.
- Write tools require valid token; invalid token returns auth error.
- `submit_candidate_claim` creates candidate; Kernel validates min_support.
- Token generation returns plaintext once; storage uses hash only.
- `pks mcp token create/list/revoke` works.
- MCP tools have test coverage.

### Web UI Completion
- Claim creation form guides user through min_support requirements.
- Claim creation submits via `submit_candidate`; claim_id auto-generated.
- Multi-dimension browsing: filter by type/status/projection/time.
- Batch accept/reject works (multi-select).
- MCP token page: list tokens, regenerate, copy.
- Claim editing: non-semantic changes apply directly, semantic changes create candidates.
- Evidence chain shows tree visualization.
- Config page can view domain policy and projection specs.
- All new endpoints have test coverage.

### Architecture
- Token generation/revocation in MCP module.
- Token permission validation in Kernel.
- Web UI only displays tokens (no token logic).
- All Claim writes go through `submit_candidate` (no bypass).
- All Claim edits go through `patch_projection_claim`.

## 7. Constraints

- MCP Server is a thin adapter. No business logic.
- Agent behavior constrained by Kernel (min_support, review, conflict detection). No separate policy layer.
- Token is the only auth mechanism. No per-agent fine-grained permissions in MVP.
- Web UI Claim creation/editing goes through Kernel interfaces (not direct storage writes).
- Core capabilities (Claim/Capsule/MD CRUD) live in Kernel. CLI is the base interface. Web UI and MCP are external adapters.
- No new heavy dependencies beyond MCP SDK.

## 目的

P3 通过受控的 MCP 接口实现 **Agent 协作**，并完善 **Web UI 管理界面**。

P2 交付了自动维护和最小 Web UI。P3 要：
- 让 Agent 通过 MCP 标准协议查询和提交知识
- 给人类一个完整的管理界面（批量审核、Claim 新建、多维度浏览、配置管理）

## 1. P3 范围

P3 聚焦 **MCP Agent 接入 + Web UI 完善**：

- MCP Server：暴露 Kernel 读写方法为 MCP 工具
- MCP Auth：基于 token 的读写权限（MCP 模块生成管理，Web UI 展示 regenerate/copy）
- Web UI 完善：Claim 新建表单、批量审核、多维度 Claim 浏览、Claim 编辑、证据链可视化、配置管理、token 展示
- Kernel token 权限：token 权限校验在 Kernel（不在 MCP 适配层）

P3 不做：
- Task Contract Engine（PKS 不是项目管理工具）
- Per-agent 细粒度权限（MVP 只需 token 级读写分离）
- 多用户协作
- Obsidian/Notion 集成（未来适配层工作）
- SQLite 索引（等查询性能需要时再加）
- 定时维护 daemon

## 2. 已定决策

- MCP Server 是 Kernel 的薄适配层，不含业务逻辑。
- Agent 行为由 Kernel 现有机制约束（min_support、review、冲突检测）。MCP 不加独立策略层。
- MCP 认证：只读工具开放，可写工具需要有效 token。
- Token 生成/撤销由 MCP 模块管理。Token 权限校验在 Kernel。
- Web UI 展示 token（regenerate/copy 按钮），不管理 token 逻辑。
- Web UI Claim 新建走 `submit_candidate`（复用现有 review 流程）。
- Web UI Claim 编辑走 `patch_projection_claim`（语素变更 → 候选，非语素 → 直接更新）。
- Claim ID 由 Kernel 自动生成，Web UI 不暴露 claim_id 输入。
- 不引入 MCP SDK 之外的重依赖。
