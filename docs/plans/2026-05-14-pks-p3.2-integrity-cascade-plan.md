# English Version

## Purpose

P3.2 strengthens knowledge integrity by detecting two failure modes that current maintenance cannot catch:

1. **Support chain breakage**: A Claim's supporting_claims reference has been superseded/expired/disputed, but the Claim itself remains accepted.
2. **Evidence source mutation cascade**: A file referenced by evidence has changed (detected via `sync_project`), invalidating not only the directly affected Claim but potentially all Claims that depend on it.

When either condition is detected, affected Claims and their dependents are marked for re-verification — not automatically rejected, but flagged for human review.

## 1. P3.2 Scope

- Support chain breakage detection in `pks maintain`
- Evidence source mutation detection (file changed → Claim needs re-verification)
- Cascade propagation: if Claim X needs re-verification, all Claims citing X as `supporting_claims` also need re-verification
- "Pending re-verification" state (computed, not a new ClaimStatus)
- Web UI display of re-verification queue
- CLI `pks maintain` reports re-verification candidates

### Not in P3.2

- Automatic re-verification (human decides)
- Semantic change detection (only file-level change detection)
- Auto-reject or auto-expire affected Claims

## 2. Decisions

- Re-verification is a **computed flag**, not a new `ClaimStatus`. Claims remain `accepted` but are flagged in health reports and Web UI.
- Cascade propagation is **upward only**: if a factual Claim is flagged, all inference/preference/constraint Claims that cite it are also flagged.
- Cascade stops at Claims that have been re-verified after the source change (their `last_verified` is newer than the change timestamp).
- `sync_project` already detects changed files. P3.2 connects changed files → affected evidence → affected Claims → dependent Claims.
- Re-verification is resolved by human action: either confirm (update `last_verified`) or supersede/expire the Claim.

## 3. Detection Logic

### 3.1 Support Chain Breakage

```text
For each accepted Claim with supporting_claims:
  For each supporting_claim reference:
    Load the referenced Claim
    If referenced Claim status is superseded/expired/disputed:
      → Flag current Claim as "support chain broken"
      → Include in re-verification queue
```

### 3.2 Evidence Source Mutation

```text
After sync_project returns changed_paths:
  For each changed file path:
    Find all accepted Claims with evidence.source_ref matching that path
    For each matched Claim:
      If Claim.last_verified < file change timestamp (last_synced_commit time):
        → Flag Claim as "evidence source changed"
        → Include in re-verification queue
```

### 3.3 Cascade Propagation

```text
Given a set of flagged Claims (from 3.1 or 3.2):
  Repeat until no new Claims are added:
    For each accepted Claim not yet flagged:
      If any of its supporting_claims references a flagged Claim:
        If Claim.last_verified < flag timestamp:
          → Flag this Claim as "dependent on unverified support"
          → Add to flagged set
```

Cascade is bounded: it only propagates upward through `supporting_claims` references, and stops at Claims whose `last_verified` is newer than the trigger event.

## 4. Data Model

### 4.1 ReVerificationIssue

```python
class ReVerificationIssue(BaseModel):
    claim_id: str
    reason: str          # "support_chain_broken" | "evidence_source_changed" | "cascade_dependency"
    trigger_claim_id: str | None = None   # The Claim that caused cascade
    trigger_source: str | None = None     # The file path that changed
    detected_at: datetime
```

### 4.2 HealthReport Extension

Add to existing `HealthReport`:
```python
class HealthReport(BaseModel):
    ...
    reverification_needed: int = 0
    reverification_issues: list[ReVerificationIssue] = Field(default_factory=list)
```

### 4.3 ClaimHealth Extension

Add to existing `ClaimHealth`:
```python
class ClaimHealth(BaseModel):
    ...
    needs_reverification: bool = False
    reverification_reason: str | None = None
```

## 5. Resolution Flow

Human resolves re-verification through Web UI or CLI:

```text
Claim flagged for re-verification
  ↓
Human reviews in Web UI (re-verification queue)
  ├── "Still valid" → update last_verified to now (Kernel method)
  ├── "Needs update" → edit Claim (patch_projection_claim, semantic → new candidate)
  └── "No longer valid" → expire or supersede
```

### 5.1 Kernel Method

New method: `verify_claim(project_id, claim_id)`
- Updates `last_verified` to today
- Writes Audit Claim
- Clears re-verification flag for this Claim (and cascaded dependents if their only trigger was this Claim)

### 5.2 CLI

```bash
pks maintain <project_id>                    # Now also reports re-verification issues
pks claim verify <project_id> <claim_id>     # Mark Claim as re-verified
pks health <project_id>                      # Shows reverification_needed count
```

### 5.3 MCP Tools

New MCP tools (write, token required):

| Tool | Kernel Method | Description |
|------|---------------|-------------|
| `verify_claim(token, project_id, claim_id)` | `verify_claim` | Confirm Claim still valid after source change |
| `get_reverification_issues(project_id)` | `health_check` → filter | List Claims needing re-verification |

Agent use case: Agent detects a file changed but confirms the Claim's conclusion still holds → calls `verify_claim` to update `last_verified`. This is audited.

### 5.4 Web UI

- Health dashboard shows re-verification count
- New section in project detail: "Needs Re-verification" list
- Each item shows: claim_id, reason, trigger source, dependent chain
- Actions: "Confirm valid" / "Edit" / "Expire"

## 7. Acceptance Criteria

- `pks maintain` detects support chain breakage (supporting Claim superseded/expired/disputed).
- `pks maintain` detects evidence source mutation (file changed after last_verified).
- Cascade propagation flags dependent Claims upward through supporting_claims.
- Cascade stops at Claims with `last_verified` newer than trigger event.
- `pks claim verify` updates `last_verified` and writes Audit Claim.
- HealthReport includes `reverification_needed` count and issue list.
- Web UI shows re-verification queue with reason and trigger info.
- Web UI provides "Confirm valid" / "Edit" / "Expire" actions.
- Affected Claims remain `accepted` (not auto-rejected).
- Tests cover: breakage detection, mutation detection, cascade propagation, cascade stop condition, verify resolution.

## 8. Constraints

- Re-verification is a computed flag, not a persisted status. It's recalculated on each `health_check` or `maintain` run.
- No automatic rejection. Human always decides.
- Cascade is upward only (through supporting_claims). It does not propagate downward.
- Detection is file-level (path changed), not semantic-level (content meaning changed).
- `verify_claim` is the only way to clear the flag (updating last_verified).

## 9. Testing Plan

### Unit tests
- Support chain breakage: Claim A supports B, A gets superseded → B flagged
- Evidence mutation: file changes after sync → Claim flagged
- Cascade: A flagged → B (cites A) flagged → C (cites B) flagged
- Cascade stop: C has last_verified after trigger → C not flagged
- Verify resolution: verify B → B unflagged, C re-evaluated

### Integration tests
- Full flow: sync → detect → maintain report → verify via CLI
- Web UI: re-verification queue display + confirm action

# 中文版

## 目的

P3.2 通过检测两种当前维护无法捕获的失效模式，强化知识完整性：

1. **支撑链断裂**：Claim 的 supporting_claims 引用已被 superseded/expired/disputed，但 Claim 本身仍为 accepted。
2. **Evidence 源头变更级联**：evidence 引用的文件已变更（通过 `sync_project` 检测），不仅直接受影响的 Claim 需要重新验证，依赖它的上层 Claim 也需要。

检测到上述情况时，受影响的 Claim 及其依赖者被标记为"待重验证"——不自动 reject，而是等待人类审核。

## 1. P3.2 范围

- `pks maintain` 中的支撑链断裂检测
- Evidence 源头变更检测（文件变更 → Claim 需重验证）
- 级联传播：如果 Claim X 需要重验证，所有引用 X 作为 supporting_claims 的 Claim 也需要
- "待重验证"状态（计算属性，不是新的 ClaimStatus）
- Web UI 展示重验证队列
- CLI `pks maintain` 报告重验证候选

### 不在 P3.2 范围

- 自动重验证（人类决定）
- 语义级变更检测（只做文件级变更检测）
- 自动 reject 或 expire 受影响 Claims

## 2. 已定决策

- 重验证是**计算标记**，不是新的 `ClaimStatus`。Claims 保持 `accepted`，但在健康报告和 Web UI 中被标记。
- 级联传播**仅向上**：如果 factual Claim 被标记，引用它的 inference/preference/constraint 也被标记。
- 级联在 `last_verified` 晚于变更时间的 Claim 处停止。
- `sync_project` 已能检测文件变更。P3.2 连接：变更文件 → 受影响 evidence → 受影响 Claims → 依赖 Claims。
- 重验证由人类操作解决：确认仍有效（更新 `last_verified`）、编辑、或 expire/supersede。

## 3. 检测逻辑

### 3.1 支撑链断裂

```text
对每条有 supporting_claims 的 accepted Claim：
  对每条 supporting_claim 引用：
    加载被引用 Claim
    如果被引用 Claim 状态为 superseded/expired/disputed：
      → 标记当前 Claim 为"支撑链断裂"
      → 加入重验证队列
```

### 3.2 Evidence 源头变更

```text
sync_project 返回 changed_paths 后：
  对每个变更文件路径：
    查找所有 evidence.source_ref 匹配该路径的 accepted Claims
    对每条匹配的 Claim：
      如果 Claim.last_verified < 文件变更时间：
        → 标记为"evidence 源头已变更"
        → 加入重验证队列
```

### 3.3 级联传播

```text
给定一组被标记的 Claims（来自 3.1 或 3.2）：
  重复直到没有新 Claim 被加入：
    对每条尚未被标记的 accepted Claim：
      如果其 supporting_claims 引用了任何被标记的 Claim：
        如果 Claim.last_verified < 标记时间：
          → 标记为"依赖未验证的支撑"
          → 加入标记集合
```

级联有界：只通过 supporting_claims 向上传播，在 `last_verified` 晚于触发事件的 Claim 处停止。

## 7. 验收标准

- `pks maintain` 检测支撑链断裂（supporting Claim 已 superseded/expired/disputed）。
- `pks maintain` 检测 evidence 源头变更（文件在 last_verified 之后被修改）。
- 级联传播通过 supporting_claims 向上标记依赖 Claims。
- 级联在 `last_verified` 晚于触发事件的 Claim 处停止。
- `pks claim verify` 更新 `last_verified` 并写 Audit Claim。
- HealthReport 包含 `reverification_needed` 计数和问题列表。
- Web UI 展示重验证队列（原因 + 触发信息）。
- Web UI 提供"确认有效" / "编辑" / "过期"操作。
- 受影响 Claims 保持 `accepted`（不自动 reject）。
- 测试覆盖：断裂检测、变更检测、级联传播、级联停止条件、验证解决。

## 8. 约束

- 重验证是计算标记，不是持久化状态。每次 `health_check` 或 `maintain` 时重新计算。
- 不自动 reject。人类始终决定。
- 级联仅向上（通过 supporting_claims），不向下传播。
- 检测是文件级（路径变更），不是语义级（内容含义变更）。
- `verify_claim` 是清除标记的唯一方式（更新 last_verified）。
