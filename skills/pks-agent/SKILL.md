# PKS â€” Personal Knowledge State

## Core Purpose

PKS manages project knowledge state so that Agents and humans can quickly grasp a project without reading all source code and documentation:
- Current project status and goals
- Key architecture decisions and technology choices
- Style preferences and conventions
- Constraints and prohibitions
- Lessons learned and known pitfalls

All knowledge is traceable â€” every Claim links to concrete evidence sources (code files, documents, conversation records, etc.).

## Structural Principles

PKS has three core layers:

```
Evidence (source reference + excerpt verbatim text)
  â†‘ supports
Claim (structured knowledge unit with type and support requirements)
  â†‘ organized by
Capsule (project knowledge container)
  â†‘ projected as
Markdown (PKS.md / PKS_PROJECT.md / journal.md / domain MD / custom MD)
```

- **Evidence**: Source proof for knowledge. Contains `source_ref` (source path) and `excerpt` (verbatim text copied from the source file â€” used by PKS to automatically verify whether the source is still valid).
- **Claim**: Atomic knowledge unit. Composed of subject/predicate/object (semantic triple), plus content (human-readable description) and evidence.
- **Capsule**: Project knowledge container. Organizes Claims by capsule_type and domain, determines which default Markdown projections are generated.
- **Markdown Projections**: Readable mappings of Claims. Each MD file is defined by a ProjectionSpec (projection rule) that filters, sorts, and renders from the Claim collection. PKS.md is the aggregation of all projections.

**Key principle**: All PKS-generated Markdown is a projection of Claims, not hand-written files. Modifying knowledge requires submitting Claims; Markdown is automatically regenerated.

## Collaboration Model

| Role | Interface | Operations |
|------|-----------|------------|
| Agent | MCP | Query context, submit candidate Claims, submit projection rules/content, verify Claims |
| Human | Web UI (localhost:8420) | Review candidates, manage Claims, edit projection rules, generate MCP tokens, view health |
| Human/Agent | CLI | Debug/maintenance, project creation, snapshot management |

**Flow**: Agent submits Claim â†’ Kernel validates min_support â†’ enters candidate queue â†’ human reviews in Web UI (accept/reject) â†’ accepted Claims automatically update Markdown projections.

**Token**: MCP write operations require a token. Tokens are generated by humans in Web UI (localhost:8420/tokens), then provided to the Agent. Without a token, Agent can only read.

| Timing | What to do |
|--------|-----------|
| **New project setup** | Multi-turn conversation with human to clarify requirements and goals, then create Capsule + base Claims + custom MD |
| **New fact discovered** | Found code structure, tech choice, module relationship â†’ submit factual Claim |
| **User states an inference** | User makes a judgment based on facts â†’ submit inference Claim (cite factual as support) |

## New Project Setup

When starting a new project, **do not create a Capsule immediately**. First have a multi-turn conversation with the human:

1. **Clarify project positioning**: What is the project? What problem does it solve? Who are the target users?
2. **Determine technical boundaries**: Tech stack, architecture style, key constraints
3. **Identify current stage**: MVP? In development? Maintenance?
4. **Plan knowledge structure**: Beyond default MD (PKS_PROJECT.md, journal.md, domain MD), are custom MDs needed?

After confirmation, create:

```bash
# 1. Initialize PKS home (first time only)
pks init-home

# 2. Create Capsule
pks new <project-id> \
  --name "Project Name" \
  --capsule-type SoftwareCapsule \
  --domain dev \
  --stage "Development" \
  --current-goal "Current goal" \
  --project-path "/path/to/project" \
  --constraints "constraint1,constraint2" \
  --yes

# 3. Generate initial projections
pks project projection <project-id> --write

# 4. Start MCP
./start.sh mcp
```

Available capsule types: `SoftwareCapsule`, `PluginCapsule`, `ArticleCapsule`, `VideoCapsule`, `GameCapsule`, `DisciplineCapsule`, `ModelCapsule`

Available domains: `dev`, `content`, `research`

## Markdown Projection Management

### Default Projections

Each capsule_type has default Markdown projections (defined by ProjectionSpec):

| capsule_type | Default projections |
|--------------|---------------------|
| SoftwareCapsule | PKS_PROJECT.md, journal.md, architecture.md, tasks.md |
| ArticleCapsule | PKS_PROJECT.md, journal.md, outline.md, facts.md |
| ResearchCapsule | PKS_PROJECT.md, journal.md, terminology.md, hypotheses.md |

### Custom Projections

When default projections are insufficient, create custom MDs:

1. Define a ProjectionSpec (specify filters: which tags/types/predicates of Claims enter this MD)
2. When submitting Claims, use corresponding tags (ensure Claim appears in target MD)
3. PKS automatically renders the MD based on ProjectionSpec

**Example**: Project needs a `security.md` to aggregate security knowledge:
- Create ProjectionSpec: `filters.tags = ["security", "vulnerability", "auth"]`
- Submit security-related Claims with `tags: ["security"]`
- PKS automatically renders these Claims into `security.md`

### Claim-to-MD Relationship

- Each Claim's `tags` and `predicate` determine which MDs it appears in
- A Claim can appear in multiple MDs (if it matches multiple ProjectionSpec filters)
- Modifying MD content = modifying corresponding Claims (via `submit_candidate_claim` or `patch_projection_claim`)

## Submitting Claims

### Basic Format

```json
{
  "subject": "PKS Kernel",
  "predicate": "uses_pattern",
  "object": "composition over inheritance",
  "content": "PKS Kernel uses composition â€” ProjectRegistry composes PolicyManager, TasteManager, etc.",
  "type": "factual",
  "tags": ["architecture"],
  "confidence": 0.9,
  "evidence": [
    {
      "source_ref": "src/pks/kernel/capsule/registry.py",
      "source_type": "file",
      "relation": "supports",
      "excerpt": "self.policy = PolicyManager(self.domains_dir)"
    }
  ]
}
```

- `claim_id`: Do not provide â€” PKS auto-generates it
- `content`: Required. Human-readable full description (core content source for projection MD)
- `evidence.excerpt`: Verbatim text copied from the source file. PKS uses it to automatically verify whether the source is still valid â€” if this text can no longer be found in the source file, the Claim is flagged for re-verification
- `tags`: Determine which MD projections the Claim appears in

### min_support Requirements

| Claim type | Evidence required | Supporting Claims required | Total support |
|------------|-------------------|---------------------------|---------------|
| `factual` | â‰Ą 1 | 0 | â‰Ą 1 |
| `inference` | â‰Ą 0 | â‰Ą 0 | â‰Ą 1 (combined) |
| `preference` | â‰Ą 1 (must include manual/conversation source) | â‰Ą 0 | â‰Ą 1 |
| `constraint` | â‰Ą 1 | â‰Ą 1 | â‰Ą 2 |

### Type Hierarchy

Higher levels can cite lower levels as support; lower cannot cite higher:
- `inference` can cite `factual`
- `preference` can cite `factual` or `inference`
- `constraint` can cite `factual`, `inference`, or `preference`

## Re-verification

When source files change or supporting Claims become invalid, PKS flags affected Claims as "needs re-verification".

- Call `get_reverification_issues(project_id)` to see the re-verification queue
- If Claim still holds: call `verify_claim(token, project_id, claim_id)` to confirm
- If Claim no longer holds: submit a new candidate Claim to supersede the old one, or leave for human review

## Rules

1. **Never access `~/.pks/` directly** â€” use MCP tools only
2. **Never edit PKS.md** â€” it is auto-generated from Claims
3. **Always provide evidence** â€” source_ref + excerpt
4. **Read context before working** â€” call `get_project_context`
5. **Respect the review process** â€” submissions become candidates; humans decide acceptance
6. **Excerpts must be exact** â€” copy verbatim from source files, do not paraphrase or abbreviate

## Fallbacks

- MCP unavailable: inform human, provide `./start.sh mcp` configuration instructions
- No write token: inform human to generate a token from Web UI (localhost:8420/tokens) and provide it to you
- Cannot satisfy min_support: explain what support is missing, ask human to provide evidence or supporting Claims
